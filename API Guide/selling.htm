Getting Started

What Are Amazon Web Services for Sellers?

Amazon Web Services for Sellers provides a secure and automated method to upload inventory details to Amazon.com (zShops and Amazon Marketplace) on a scheduled basis. It also enables the automation of other important seller activities such as generating order reports and issuing bulk refunds. This is a first step toward real-time updates that will enable selling partners to better manage their inventory in multiple physical and virtual locations.

This project utilizes HTTPS (Hypertext Transfer Protocol Secure), a protocol that is designed to transfer encrypted information between computers over the World Wide Web. HTTPS is a unique protocol that combines SSL and HTTP to provide secure data transmission. The goal of this document is to help our valued partners understand the technical requirements of our HTTPS solution.

Getting Registered

In order to sell products on the Amazon Web site, you’ll first need to register with Amazon.com.  To get started, see the help information contained on the Amazon Web site.

Security

Use of HTTPS ensures that all data uploads and downloads will be encrypted and secure.

Sellers will need to use SSL (Secure Sockets Layer) and PKI (Public Key Infrastructure) technology to communicate securely with Amazon.com—if you are using Perl, note that both of these technologies are built into Perl to ensure that scripts written to automate the data communication are correctly configured.

The identity of the seller will be verified using the "Basic HTTP Authentication Scheme." The seller will provide his or her e-mail address and password in the HTTPS header, encoded using base64. 

Automation

Please note that the below instructions are detailed for partners using Perl. However, you may use any language.

Script Configuration

In order to automate the secure uploading and retrieval of files and/or reports, selling partners will need to create an HTTPS client. This client will consist of scripts that automate the uploading and retrieval of files.

  • Upload: To upload a file, the seller uses a Perl script that they have written to send a POST request with the file attached in the request body. The seller should include both their Amazon.com e-mail address and password in the request header.
  • Retrieve: To retrieve a file, the seller sends a POST request for the particular file. Amazon.com sends the requested file in an HTTPS response, at which point the seller would redirect the response to a specified directory and then save the file.
  • Notifications: Selling partners who participate in the HTTPS solution can have the automatic e-mail notifications from Amazon.com regarding uploads and reports turned off, relying instead on the HTTPS responses in XML.

If an upload fails due to internal error, Amazon.com will send e-mail with the batch status and information.

Types of Requests (Detailed)

Sellers will be able to manage their inventory at Amazon.com using the following request types:

  • Add/Modify/Delete
  • Modify/Delete
  • Purge and Replace

The difference between each of these request types is explained below.

Inventory Upload via Add/Modify/Delete

Use Case: A seller should use the Add/Modify/Delete request to upload a batch of Amazon Marketplace and/or zShops listings containing additions, modifications, and/or deletions to their current inventory. Deletions must be marked as such in their template, (with a d in the add-delete field) and our system will update changed items to reflect the modifications.

A seller should choose the Add/Modify/Delete option when they have a large amount of inventory to upload onto the Amazon.com platform.

HTTPS POST request to the following URL:

https://secure.amazon.com/exec/panama/seller-admin/catalog-upload/add-modify-delete

Header Content:

·         Username (using basic authentication)

·         Password (using basic authentication)

·         Developer Token

·         Choice of program type

o        Amazon Marketplace only (for UIEE users) (default)

o        zShops only

o        Amazon Marketplace and/or zShops

·         Choice of file format

o        Tab-delimited (default)

o        UIEE

Message Body:

The body of the message should include the actual inventory file desired for upload with the required Amazon.com headers and values. These required values can be found via the URL below for program type "zShops only":

http://s1.amazon.com/exec/varzea/subst/seller-admin/fixed-price-template-fields.html

and via the URL below for program type "Amazon Marketplace and/or zShops":

http://s1.amazon.com/exec/varzea/subst/seller-admin/marketplace-template-fields.html

Inventory Upload via Modify/Delete

Use Case: A seller should use the Modify/Delete request when they want to upload modifications to their current inventory but don't want to supply all of the fields that an Add/Modify/Delete file requires. The field "sku" is required in this upload file, as well as at least one additional field that they are choosing to modify. The allowable fields of modification in the Modify/Delete Only option are price and quantity. Please note: to submit a delete, use the quantity column and enter "0" as the value.

HTTPS POST request to the following URL:

https://secure.amazon.com/exec/panama/seller-admin/catalog-upload/modify-only

Header Content:

·         Username (using basic authentication)

·         Password (using basic authentication)

·         Developer Token

·         Choice of program type (UploadFor)

o        Amazon Marketplace only (for UIEE users) (default)

o        zShops only

o        Amazon Marketplace and/or zShops

·         Choice of file format (FileFormat)

o        Tab-delimited (default)

o        UIEE

Inventory Upload via Purge and Replace

Use Case: A seller should use the Purge and Replace request when they want to completely purge all of their current Amazon Marketplace and/or zShops listings and replace them with a new set of offerings contained in their template.

Types of Requests (Detailed)

Once a seller has uploaded their inventory on the Amazon.com platform, they may want to check the status of their upload. Each inventory upload (whether it is an Add/Delete/Modify, Delete/Modify or Purge/Replace) has a batch ID associated with it. Sellers can either check the status of multiple batches, or retrieve error logs and/or quick fix files for specific batch IDs.

Checking Upload Status         

Use Case: A seller should use the Upload Status request when they want to find out the status of their latest uploads. (They will need to specify the number of uploads that they want to check on in the request header). Each upload (or “batch”) will return a batch ID, status report, and other information. If the status of a particular batch is negative, the seller can use the batch ID to retrieve an error log or quick fix file. (See below).

HTTPS POST request to the following URL:

https://secure.amazon.com/exec/panama/seller-admin/catalog-upload/get-batches

Header Content:

·         Username (using basic authentication)

·         Password (using basic authentication)

·         Developer Token

·         Number of Batches (default=10)

Retrieval of Inventory Upload Error Log

Use Case: If something goes wrong with a specific inventory upload, the seller can retrieve the upload’s Error Log to determine what caused the problem. The Error Log will contain the row number of any applicable error or warning, the SKU, and the item name. It will also contain the message type (status message, data error, template error, etc.) and the message itself. The seller can use this information to fix any errors and re-upload those rows.

HTTPS POST request to the following URL:

https://secure.amazon.com/exec/panama /seller-admin/download/error_log

Header Content:

·         Username (using basic authentication)

·         Password (using basic authentication)

·         Developer Token

·         Batch ID

Retrieval of Inventory Upload Quick Fix File

Use Case: A seller can also retrieve a Quick Fix file if something goes wrong with a particular upload. A Quick Fix file is similar to an Error Log, except that it contains only the error rows that did not get uploaded to the site due to missing required information or inappropriate information.

HTTPS POST request to the following URL:

https://secure.amazon.com/exec/panama /seller-admin/download/quick_fix

Header Content:

·         Username (using basic authentication)

·         Password (using basic authentication)

·         Developer Token

·         Batch ID

Checking Status of Order Fulfillment Reports

Use Case: A seller should use the Report Status Request when they want to check on the status of multiple order reports. They should include the number of reports that they want to check on in the request header.

HTTPS POST request to the following URL:

https://secure.amazon.com/exec/panama/seller-admin/manual-reports/get-report-status

Header Content:

·         Username (using basic authentication)

·         Password (using basic authentication)

·         Developer Token

·         Number of Reports (default = 10)

·         Choice of Report Name

Retrieving Order Fulfillment Reports

Use Case: A seller should use the Order Fulfillment Request when they want to download a specific report. The request header must include either the ID of the report in question, or a generic report name (Order, Preorder, etc.). If a generic report name is used, then the system will retrieve the latest available report.

HTTPS POST request to the following URL:

https://secure.amazon.com/exec/panama/seller-admin/download/report

Header Content:

·         Username (using basic authentication)

·         Password (using basic authentication)

·         Developer Token

·         Report ID or Choice of Report Name

Uploads and Order Fulfillment Report Logistics

Schedule

File processing will be restricted to one file at a time. For example, if a seller has a scheduled upload at the same time he or she schedules a delete file pickup, the requests will be queued in the order received so that only one request will be processed at a time.

If a purge and replace request fails, the remaining files in the queue will continue to be processed. To avoid data corruption, it will be the sellers’ responsibility to submit a fresh purge and replace request in the next schedule. If it is urgent that a purge and replace file is processed before the next scheduled pickup, the seller can instead use the manual upload found using the following URL:

https://secure.amazon.com/exec/panama/seller-admin/catalog-upload/catalog-upload-now/

  • The POST requests for inventory upload (Add/Modify/Delete) should be sent every four hours or multiples of four hours (12 midnight, 4a.m., 8a.m., 12 noon, 4p.m., 8p.m. daily) All times PST
  • The POST requests associated with the purge and replace should be either sent every 12 midnight or every Friday 12 midnight . If the purge and replace file is going to be over 20 MB, it should only be scheduled for pickup once a week.
  • The POST requests associated with the upload of delete can be sent as frequent as every 30 minutes or multiples of 30 minutes.

Purge Policy

·         Pre-order reports and open listings are saved for 10 days.

·         All other reports are saved for 30 days.

Status Notifications

 Email

We will not send e-mail notifying successful pick up because the seller will receive an automatic response as part of the HTTPS protocol.

E-mail will be sent if an upload batch fails do to an internal error. This e-mail will include information regarding the number of records, if any, that were processed before the batch failure and will instruct the seller to resubmit the file.

Failure/Edge Case Notification

Login Failure

When login and password fails for any request, the HTTPS response will include a 404 error that notifies the seller that there was an error on the client’s end.

Business Logic Validation Failures

There are two levels of business logic failures

  1. When a request is submitted.  If a request fails at submission time, the error will be returned in the HTTPS response.
  2. When the request is processed

If the request fails during processing (no records are valid) the error(s) will be noted in the error log file, which can be downloaded as explained in the previous sections.

File Fatal Processing

Because the file processing is done asynchronously, if a processing fatals, Amazon.com will send e-mail stating that the request caused a fatal error and that the file should be resubmitted.
